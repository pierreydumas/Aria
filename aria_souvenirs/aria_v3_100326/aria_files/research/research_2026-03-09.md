## Key Findings

### 1. LiteLLM Model Management Architecture

LiteLLM Proxy provides a sophisticated model management system that supports:

- **Dynamic Model Addition**: Add models via `/model/new` API without restarting the proxy
- **Model Information Endpoint**: `/model/info` returns comprehensive details including:
  - Max tokens per model
  - Cost per input/output token
  - Custom metadata (descriptions, labels, display names)
  - Automatically pulls from LiteLLM's cost map: `model_prices_and_context_window.json`

**Relevance for Token Router:** The cost map can serve as a reference for pricing tiers when building task-to-model routing decisions.

### 2. Load Balancing & Routing Strategies

LiteLLM offers multiple routing strategies for distributing requests:

| Strategy | Description |
|----------|-------------|
| `simple-shuffle` | Default; distributes randomly across deployments |
| `least-busy` | Routes to deployment with lowest current load |
| `usage-based` | Routes based on token usage patterns |
| `latency-based` | Routes to fastest responding deployment |

**Key Insight:** RPM (requests per minute) and TPM (tokens per minute) can be set per deployment for capacity-aware routing. These can become **hard limits** with `enforce_model_rate_limits: true`.

### 3. Model Priority & Fallbacks

- **Deployment Ordering**: Use the `order` parameter to prioritize specific model instances
- **Fallback Chain**: Multiple deployments with the same model alias create automatic fallback behavior
- **Redis Integration**: For multi-instance deployments, Redis shares rate limit state across proxy instances

### 4. Practical Implementation Patterns

**Config Structure Example:**
```yaml
model_list:
  - model_name: gpt-3.5-turbo
    litellm_params:
      model: azure/gpt-3.5-turbo
      api_base: https://...
      rpm: 60  # Requests per minute capacity
      tpm: 120000  # Tokens per minute capacity
    model_info:
      description: "Azure GPT-3.5 for production"
      order: 1  # Priority ordering
```

## Insights for Token Budget Router Implementation

### Task Complexity Classification Approach
Based on this research, the Token Budget Router should:

1. **Analyze Prompt Characteristics**:
   - Token length (estimated via `tiktoken` or similar)
   - Task keywords (code, creative, analysis, conversation)
   - Expected output format (code blocks, prose, structured data)

2. **Tier-Based Routing Logic**:
   - **Free Tier (qwen3-mlx)**: Simple conversations, short queries, casual tasks
   - **Medium Tier (trinity-free)**: Moderate complexity, creative writing, summaries
   - **Premium Tier (kimi)**: Code generation, complex analysis, long-form content

3. **Budget Tracking**:
   - Track cumulative token spend per session/hour
   - Implement threshold alerts when approaching limits
   - Consider TPM/RPM limits from LiteLLM config as routing constraints

4. **Fallback Integration**:
   - Hook into LiteLLM's existing fallback chain
   - If premium model fails (rate limit, timeout), downgrade to next tier
   - Maintain quality scoring per tier to validate routing decisions

### Architectural Recommendation

The `TaskComplexityAnalyzer` should calculate a **complexity score (0-100)** based on:
- Prompt length (20% weight)
- Vocabulary density/code presence (30% weight)
- Task category detection via keyword matching (30% weight)
- Historical performance for similar prompts (20% weight)

Route thresholds:
- 0-30: qwen3-mlx
- 31-60: trinity-free
- 61-100: kimi
